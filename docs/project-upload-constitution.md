# 项目上传宪法

> 这份文档是 `data/projects.ts` 内容生成/更新时必须遵守的规则。无论是人工编辑还是通过
> `upload-project` skill 自动生成，都以此为准。目的：确保每次新增项目时，脱敏尺度、
> 诚实程度、数据结构、语气风格都和已有的四个项目（`hyperliquid-btc-bot` /
> `trade-agent` / `tidehook` / `lingua-nest`）保持一致，不因为换了一次会话/换了个人
> 操作就跑偏。

## 1. 数据结构

严格遵循 `data/projects.ts` 里的 `Project` 接口，不额外发明字段，不遗漏必填字段：

- `slug`：项目的英文短标识，小写连字符分隔，取自项目名或其中最有辨识度的部分。
- `status`："shipped" 表示"这是真实内容"（区别于占位符 "todo"），**与项目实际完成度
  无关**——哪怕项目还在 dry-run、还没上线、甚至暂停开发，只要内容是真实的就用
  "shipped"。真实完成度诚实地写进 `progress.status` 的文字里。
- `featured`：默认 `true`。首页最多显示 5 个（`ProjectsSection` 的 `.slice(0, 5)`），
  超过 5 个时才需要跟用户确认哪些保留在首页、哪些进 `/projects` 归档页。
- `title` / `oneLiner` / `problem` / `method` / `result` / `tech` / `links`：沿用现有
  卡片正面字段。`result` **必须包含具体数字**，禁止"效果良好""反响不错"这类无凭据自夸。
- `direction.vision` / `direction.roadmap` / `direction.milestones[]`：抽屉"01 方向"面板。
  `milestones[].status` 用 `done` / `in-progress` / `planned` 三态**如实**反映当前进度，
  不能为了好看全标 `done`。
- `progress.status` / `progress.metrics[]`：抽屉"02 进展"面板。`status` 是一段诚实的
  当前状态描述（"目前 dry-run，未投入实盘资金""正在重启做真机测试"这类），`metrics`
  是可验证的量化数字。
- `skills`：抽屉"03 技能"面板，标签化的技能/工具/方法论，中英文可以混排（如
  `"零信任数据设计"`、`"Prompt Engineering"`），不强制每个词条都翻译。
- 每个 `Localized` 字段必须同时给出 `en` 和 `zh`，两版语气一致但不是字面直译——要符合
  各自语言的表达习惯，且都要贴合站点"Quiet Terminal"的克制、技术性语气，不用营销辞令。
- 私有仓库：`links.demo` / `links.github` 留空（`undefined`）。只有确认是公开仓库
  （比如网站自己）才给链接。

## 2. 脱敏红线（不可协商）

生成内容前，必须先假设源材料（README、内部文档、代码、commit history）里可能混有
以下内容，逐项检查并清除：

- **真实公司/客户名称**——一律匿名化描述业务性质，不点名（参考 `trade-agent` 项目：
  真实客户是"深圳市涌序智能供应链有限公司"，最终文案只写"a B2B export lead-gen
  pipeline"）。
- **联系方式**——电话、微信号、WhatsApp、邮箱等一律不出现。
- **基础设施地址**——服务器/VPS IP、内网地址、数据库连接串一律不出现。
- **密钥类信息**——API key、token、密码、钱包私钥/助记词，哪怕已经在源文档里打了码
  也要重新确认没有漏网的。
- **他人隐私**——涉及伴侣、家人等真实第三人隐私内容的项目/文章，参考站内
  `siddhartha-traders-samsara`（悉达多的河流）先例：**必须先获得当事人同意**才能上线，
  自动化流程遇到这类信号要停下来问人，不能自己决定发布。
- **商业机密**——合作方的具体定价、合同条款、未公开的商业策略，只提炼可公开的技术/
  方法论层面信息。

拿不准某条信息算不算敏感时，按更保守的方向处理（先脱敏/先问用户），不要自己判断"应该
没事"。

## 3. 检索与提炼

- 优先读取项目自身的说明文档：`README.md`、`CLAUDE.md`、`PROJECT_BRIEF.md`、
  `PROGRESS.md`、架构/设计文档等——这些通常已经是项目作者对自己项目的第一手、结构化的
  描述，比重新分析代码更可靠、更省 token。
- 文档不够时再读关键代码/配置，找可验证的量化数字（测试结果、性能数字、处理量等）。
- 信息不足、有歧义，或者判断不出这条内容该不该露出时——**问用户**，不要编造或猜测着
  写一个"看起来合理"的版本。

## 4. 写入与验证流程

1. 编辑 `data/projects.ts`，新增或更新对应 `Project` 条目。
2. `npm run build && npm run lint`，必须两者都干净。
3. 本地起服务，用真实浏览器视觉验证：新卡片桌面端/移动端渲染正常，"+archive" 抽屉三个
   面板（方向/进展/技能）内容正确显示，双语都过一遍。
4. 确认无 console 报错。
5. `git commit`（提交信息用英文，遵循仓库现有 commit 风格），是否 `push` 需要跟用户
   确认——这个仓库的 `main` 分支直接对应生产环境自动部署，push 即上线。

## 5. 参考案例

`data/projects.ts` 里已有的四个项目就是这份宪法的实际执行样本，新项目的文案密度、
数字使用方式、脱敏尺度都应该向它们看齐，而不是自成一派。
